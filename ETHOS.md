# Keystone Ethos

## The System Design Operating System

Keystone is not a diagram tool.
Keystone is not a prompt pack.
Keystone is the forcing function that turns architecture conversations into architecture decisions.

## Builder Philosophy

The engineer who uses Keystone does not get better diagrams.
They get better thinking — structured, trade-off-aware, failure-conscious.

The AI is the instrument.
The architect is still you.

## Core Principles

### 1. Structured Thinking
Every design passes through explicit requirements → estimation → design → decisions → risks. No hand-wavy architecture.

### 2. Trade-off Transparency
Every component choice includes "why not alternative." Every decision gets cost-tagged.

### 3. Failure Consciousness
"What breaks first at 10x load?" is mandatory. SPOFs get named before production finds them.

### 4. Token Efficiency
Caveman protocol compresses prose while preserving structure. Maximum signal, minimum waste.

### 5. Opinionated but Justified
Boring tech defaults (Postgres, Redis, etc.) unless business requirements justify exotic choices.

### 6. Cost as Architecture
Every design includes cost estimation. Architecture decisions impact runway.

### 7. Security First
Every credential flow, PII handling, and auth mechanism gets flagged in every design.

### 8. Builder-Focused
Designed for the solo CTO or technical founder. Gets you from idea to production architecture in one session.

## The Promise

Install once. Run `/keystone-arch "design X"` and get:
- Requirements document
- High-level design with Mermaid diagram
- Tech stack decisions with trade-offs
- Infrastructure sizing and costs
- Risk register with mitigations
- Architecture Decision Records

All in one session. All exportable. All production-ready.
